# Introduction to Instruction Level Parallelism
Instruction-Level Parallelism (ILP) is one of the core concepts behind modern fast processors. It is the architectural technique that allows a CPU to execute multiple instructions simultaneously, ensuring that hardware resources are pushed to their maximum potential rather than sitting idle. Every processor you interact with today, from the chip in a smartphone to a data-center server, is designed around this idea in some form.

This exhibit explores **pipelining**, the primary hardware technique CPUs use to achieve ILP. We'll walk through why it was needed, how it works, what can go wrong when instructions overlap, and how modern hardware resolves those conflicts automatically.

## Why Pipelining Exists: The Sequential Bottleneck

Before pipelining, processors executed instructions in a strictly sequential manner. Each instruction had to go through all five stages of execution—Fetch, Decode, Execute, Memory Access, and Write-back—before the next instruction could even begin. This meant that at any given moment, only one hardware unit was active while the rest sat idle, creating a severe performance bottleneck.

With non-pipelined execution, if an instruction takes 5 clock cycles to complete, a sequence of **N** instructions will take **5N** cycles. Increasing the instruction load directly multiplies the execution time, leaving the hardware units utilized only 20% of the time on average.

## What is Pipelining?

Pipelining is a technique that enables the CPU to overlap the execution of multiple instructions, much like an assembly line in a factory. Instead of waiting for one instruction to finish completely, the processor begins fetching the next instruction as soon as the current one moves to the next stage.

Modern processors divide instruction execution into five classic stages:

1. **Fetch (F):** Grab the instruction from instruction memory.
2. **Decode (D):** Figure out what the instruction means and read the required registers.
3. **Execute (E):** Perform the mathematical, logical, or address calculation in the ALU.
4. **Memory (M):** Read from or write data to data memory (if applicable).
5. **Write-back (WB):** Save the final result back to the register file.

In an ideal pipeline, a new instruction is completed every clock cycle once the pipeline is full. A sequence of **N** instructions takes only **4 + N** cycles to execute, achieving a near-5x speedup over sequential execution for large instruction counts.

## The Danger: Pipeline Hazards

Although pipelining increases performance, it introduces **hazards**—situations where the next instruction cannot execute in the designated clock cycle. Hazards force the CPU to stall or discard instructions, reducing overall throughput.

### Data Hazards

A data hazard occurs when an instruction depends on the result of a previous instruction that has not yet completed execution. For example, if a `SUB` instruction needs to read register `R1` which is currently being computed by a preceding `ADD` instruction, the pipeline must ensure the correct value is ready before `SUB` executes.

### Branch Hazards (Control Hazards)

A branch hazard occurs when the pipeline fetches conditional branch instructions (like `BEQ`). Because the target address and outcome of the branch are not known until later stages (Execute or Memory), the CPU does not know which instruction to fetch next. 

If the branch prediction guesses incorrectly, the wrong instructions will enter the pipeline. Once the misprediction is detected, those incorrect instructions must be **flushed** (turned into NOPs) to prevent them from modifying register or memory state, causing a cycle penalty.

## Solutions: Hardware Interventions

To resolve hazards, CPU hardware can inject stalls (bubbles) to delay dependent instructions. Stalls, however, waste valuable execution cycles. Modern processors implement advanced hardware optimizations to maintain peak efficiency:

1. **Data Forwarding (Bypassing):** Routes the execution result directly from the ALU output back to the ALU input for subsequent instructions, bypassing the write-back stage and avoiding stalls.
2. **Branch Prediction:** Hardware predicts whether a branch will be taken or not, fetching the predicted path to keep the pipeline full. Flushes only happen on a misprediction.

## Summary

Instruction-Level Parallelism via pipelining is the foundation of high-performance processor design. While overlapping instruction execution dramatically increases throughput, hardware designers must implement complex mechanisms like forwarding paths, interlocks, and branch predictors to manage hazards and keep the execution units running at maximum efficiency.
